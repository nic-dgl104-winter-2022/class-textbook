Model View Presenter(MVP)

An explanation in detail of how the Model View Presenter architecture might be the answer that developers seek.

MVP is an abbreviation that you have probably heard often and it stands for ‘most valuable player’, well most of the time it stands for that but not in this case, so let’s leave the jokes aside and focus on what I want to elaborate on in this article which is the MVP pattern. MVP pattern stands for model view presenter pattern. MVP it’s one of the most popular architectural patterns which divide the application into three layers: the Model, the View, the Presenter.
Note: MVP is only a derivation of MVC architecture.

History

Taligent was a software company as well as the origin of MVP. So MVP is a software pattern that was originally used by Taligent in 1990. It was used as an underlying programming model for application development in C++ based CommonPoint environment. After a while, the model was transferred from Taligent to Java.

Andy Bower and Blair McGlashan of Dolphin Smalltalk in 1998 adapted the MVP pattern to form the basis for their Smalltalk user interface framework. MVP in 2006, was incorporating in Microsoft’s documentation for user interface programming in .Net framework.

MVP Architecture Pattern

The MVP pattern is very similar to the MVC pattern. In this architecture pattern “p” stands for the presenter. So it has there parts just like MVC but here the controller is replaced by the presenter although they do not perform the same function. The Presenter can address all the UI events on behalf of the view. It takes the input from the users, proceeds the data in the Model, and then transforms the results back to the View.

MVP is known also as a compound pattern. MVP pattern is usually performed with windows forms applications and ASP.NET WebForms.

Model- is used to work with data. A model can be an interface that is responsible for accessing APIs and local database or cache so the main role of the model is managing data.

View- implements data display. It only should be responsible for displaying things and should not contain any business logic. So the view only manages and displays the pages.

Presenter- it’s known as the middle-man messenger because it implements the interaction between the model and the view. Since the model and view do not communicate directly, the presenter is used as a bridge it retrieves data from the model and then it formats in a way that the view can understand.

Pros and cons of MVP

Each pattern has its advantages and disadvantages depending on the environment where it is used.
Pros of MVP usage:
1. Code reuse- Code reusability will be provided by the separation of concern principle. The design will have a proper domain model and business logic in its logical unit.
2. Test-driven approach- the application of isolation allows the components to be tested separately. This separation of the logical part of the view which is the presenter with the visual view which is the actual view enables the facilitation of unit testing.
3. Adaptable design- Changes and additions can be much easier to apply if the design is good. Isolated code provides the freedom of choosing several views and data sources.
4. Layering- MVP separates the view logic from business logic.
Cons of MVP usage:
-Higher complexity
-Experience and knowledge makes a difference in the right implementation
-Not suitable for simple and small solutions

MVP vs MVC

I mentioned above that MVP is nothing more than a derivation of MVC so it has similarities but also has radical dissimilarities. It’s all about the dependence of each layer on the other layers or more precisely which layer has the main control over the others. in the MVP the view is in charge, while in MVC the controller is in charge
To sum up Controllers in the MVC have more control over the UI and handle events, while a presenter in the MVP is more passive and just presents information through the UI.So there isn’t much difference there is a very thin dividing line between MVP and MVC.

Conclusion

MVP is a derivation of MVC. it’s very flexible it allows adding new features and ideas at a much lower cost, which in turn enables high-quality products. Many environments support the MVP pattern like .Net, Java, Kotlin, Php.
The presentation layer and business logic are separated, ensuring that the code can be easily tested and also easily understood. It should be noted that it is not preferred to be used in small applications.
